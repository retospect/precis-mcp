DFT Modeling
Executive Summary
Constructing the atomic input to a Density Functional Theory (DFT) calculation today is a manual, script-level task. The practitioner builds a unit cell in Atomistic Simulation Environment (ASE) or pymatgen, edits coordinates by hand, hands the geometry to VASP, Quantum ESPRESSO, or CP2K, inspects the output, and iterates. The substrates around this loop have converged: the DFT engines themselves are mature, workflow layers (AiiDA, Atomate2) handle bookkeeping, and a 2025 to 2026 generation of universal machine-learning interatomic potentials (MACE-MP, PET-MAD, MatterSim, Orb) absorbs most relaxations at minute-scale cost. What remains stubbornly manual is the construction surface itself — the act of proposing, mutating, and inspecting candidate structures. Large Language Model (LLM) agents have begun to call ASE through tool wrappers, but the comparative literature is consistent that they fail on the exact geometric reasoning steps practitioners cite as friction: fractional coordinates, symmetry placement, surface termination selection, and image-atom recognition across periodic boundaries . This report designs an Model Context Protocol (MCP) server that occupies that unaddressed layer: a small, deterministic verb set that lets an LLM drive structure construction the way it drives a code editor, with all physics deferred to established engines.
The design rests on five load-bearing decisions. First, structures live in PostgreSQL as cells, atoms, and bonds with stable global handles of the form a<Element><n> (for example aPd123); the element prefix is a redundancy the LLM finds useful and the global counter prevents aliasing across cells [dc1571458, dc1572264]. Second, periodic boundary conditions are surfaced through explicit image triples [i,j,k] on bonds and on cross-cell queries, so the model never has to infer wraparound from coordinates alone . Third, the verb surface is intentionally small: insertion primitives (atom, line, sheet, cube), local mutations (move, lock, remove, set_bond), a relax call, and inspection verbs that return structured neighbour, distance, and bond-graph data rather than free text [dc1571460, dc1572456]. Fourth, the server steers the model out of raw Cartesian coordinates wherever possible, exposing named-site operations (insert_above(site=hollow_fcc), replace(atom=aH7, with='OH')) that resolve to coordinate deltas internally — directly addressing the documented LLM weakness on three-dimensional geometry . Fifth, the relaxation engine is delegated, not reimplemented: ASE provides the calculator contract, an MACE-MP, the PET-MAD family, MatterSim, and Orb are now positioned the way pretrained transformer backbones are positioned in natural language: as accurate but systematically biased priors whose remaining error can be cheaply corrected with a small in-domain dataset [citation pending]. Preparing structures for DFT calculations is the rate-limiting step of most surface and materials work today, and it has no conversational interface. Practitioners build cells in ASE, edit fractional coordinates by hand, invoke a relaxation engine, parse outputs, and iterate. The engines themselves (VASP, Quantum ESPRESSO, CP2K) have converged; universal MLIPs (MACE-MP, PET-MAD, MatterSim, Orb) have collapsed the cost of a relaxation from hours to seconds; workflow runners (AiiDA, Atomate2) have solved provenance and orchestration. The unaddressed layer is the one above all of them: a stable, deterministic surface that lets an LLM propose, mutate, and inspect periodic atomic structures the way it drives a code editor or a file system.
This report specifies an MCP server that fills exactly that gap. PostgreSQL is the single source of truth for every structure (cell, atoms, bonds, locks, provenance); files (Crystallographic Information File (CIF), POSCAR, XYZ) are generated on demand and never stored. Cells are explicit periodic boxes with first-class image addressing: an atom is aPd123, its periodic image one cell to the west is aPd123[-1,0,0], and bonds across walls carry the offset triple so the engine and the LLM agree on which image is bonded to which. Atoms expose handles, locations, lock state, valence, hybridisation, and per-neighbour bond order and bond stress. Construction is template-first: the LLM inserts a sheet, line, cube, slab, cluster, or named-site adsorbate rather than placing atoms by coordinate. Relaxation is delegated to an engine tier chosen by cost (universal MLIP for pre-relax and screening, plane-wave DFT for production), and the MCP supplies the orchestration around it (caching of converged structures, retry on tolerance failure, structured force/energy consistency checks).
Two design choices are load-bearing and carry the report. First, current LLMs reason poorly about three-dimensional coordinates and even more poorly about wraparound under Periodic Boundary Conditions (PBC); the API must therefore steer the model out of raw coordinate space whenever it can. Named-neighbour operations (view.neighbors(of=aPd123), insert_above(site=hollow_fcc), replace(atom=aH7, with='OH')) let the LLM work in the vocabulary surface chemistry already thinks in, and the server resolves names into Cartesian deltas. Second, a structured validator runs before any compute is spent: it rejects bond lengths below the sum of covalent radii, valences above the element's typical maximum, and atomic overlap below a hard-sphere cutoff. Each refusal names the rule, the offending value, and a suggested fix in the same template vocabulary the LLM proposes moves in. A check that takes microseconds blocks a failure mode that would otherwise burn minutes of GPU time per mistake.
Introduction
Constructing and iterating on atomic structures for DFT calculations today requires a sequence of manual, script-level operations: building unit cells in ASE, adjusting coordinates, invoking a relaxation engine, inspecting outputs, and repeating. No stable, conversational API exists at LLM altitude that lets a model drive this loop the way it would drive a code editor or file system. This report describes the design of an MCP server that provides exactly that substrate: a small, deterministic verb set that lets an LLM propose, mutate, relax, and inspect atomic structures for DFT preparation, in the same way a CAD MCP would let it draft and modify mechanical parts. The emphasis is on stability and auditability, not autonomy; the server exposes structure manipulation as first-class tool calls and defers all physics to established engines.
Version 1 is deliberately narrow. The element palette is restricted to Pd, Cu, C, and H, covering palladium-surface and carbon-adsorbate use cases while keeping the chemical space small enough for thorough interface validation. Structures are represented in periodic cells with explicit PBC. A bond graph is maintained alongside the coordinate set as an ergonomic aid for the LLM: topology gives the model a stable handle on connectivity that does not require inferring bonds from interatomic distances at each step. Relaxation is not implemented natively; the MCP delegates to mature engines and treats them as black-box executors that consume a structure and return a converged one. PostgreSQL is the single source of truth for all structure state. File formats (CIF, POSCAR, XYZ) are generated on demand from the stored representation rather than stored as files.
LLMs face three compounding difficulties when manipulating atomic structures. First, 3D geometric reasoning under periodicity is non-trivial: placing an atom relative to a surface site requires simultaneous tracking of fractional and Cartesian coordinates, correct application of minimum-image conventions, and avoidance of unphysical bonds across cell boundaries. Second, DFT convergence is unforgiving: small structural errors such as overlapping atoms or inconsistent cell parameters can silently produce diverged or unphysical results, and the diagnostics returned by engines assume a user who will read a full traceback. Third, no widely adopted conversational API exists for atomic structure manipulation at LLM altitude. ASE is the de facto standard in computational chemistry, but its interface is Pythonic rather than conversational, and its error surface is designed for interactive Python sessions rather than tool-call loops. Recent work has begun to quantify this gap: the AtomWorld benchmark evaluated LLMs on precise geometric operations over CIF files and found consistent failure on tasks that require applying symmetry operations, placing atoms at fractional coordinates, or detecting image atoms across periodic boundaries.
The intended workflow mirrors how an experimental surface chemist reasons: begin from a reference slab, place adsorbates at candidate sites, relax, inspect geometry and energy, then modify and repeat. The MCP exposes four primitive insertion modes corresponding to common construction idioms: single atom, sheet (a monolayer or adlayer), line (a row of atoms), and cube (a bulk block). A relax call submits the current cell to the configured engine and returns once convergence is reached. Inspection returns per-atom positions, the bond graph, and engine-reported energies in structured output the LLM can parse without writing Python. Locks are used only to prevent concurrent mutation of a cell under active relaxation; there is no broader transaction model. This loop compresses what normally requires hours of scripting into a small number of sequential tool calls, preserving the iterative reasoning pattern of experimental chemistry while eliminating the low-level file handling.
The remainder of this report is organized as follows.  surveys existing LLM-for-materials tooling and identifies the gap this server is designed to fill.  and  define the cell and atom data models, including PBC conventions and bond-graph representation.  describes the PostgreSQL schema and the import/export pipeline.  evaluates relaxation engine options and the interface contract for delegating computation.  specifies the MCP verb set, addressing conventions, and observation output format.  analyzes LLM-specific ergonomic challenges that shaped these design choices. Recommendations across the report cluster into three areas: interface design (verb granularity and the error surface the LLM sees), storage (schema for periodic structures and bond graphs), and engine coupling (async relaxation submission and convergence monitoring).
Findings
State of the art in DFT modeling and structure tooling
The production DFT ecosystem for solids is dominated by a small set of plane-wave and mixed-basis codes. VASP is commercial, plane-wave with projector-augmented-wave pseudopotentials, and remains the de facto reference for catalysis and surface science. Quantum ESPRESSO is the open-source counterpart, also plane-wave with pseudopotentials and projector-augmented-wave support, and is explicitly framed by its maintainers as a community open-source development platform on which methodological extensions are built . CP2K occupies a middle ground by combining Gaussian basis functions for the wavefunctions with an auxiliary plane-wave representation for the density; this hybrid Gaussian-plane-wave scheme is the workhorse for ab initio molecular dynamics on transition-metal surfaces where pure plane-wave runs are infeasible at the required cell sizes . GPAW, ABINIT, and FHI-aims complete the picture: GPAW is open and uses the projector-augmented-wave method on a real-space grid, ABINIT is open plane-wave, and FHI-aims is all-electron with numeric atom-centered orbitals. The all-electron versus pseudopotential split matters when modelling core-level spectroscopy or heavy elements, but for the Pd, Cu, C, and H palette targeted in this report any of the pseudopotential codes is appropriate, and the choice is usually made on licensing, parallel scaling, and the user's prior familiarity rather than physics.
Above the engines sits a thinner but more important layer: the structure-handling libraries that practitioners actually touch. ASE is the lingua franca, a Python library that defines the canonical Atoms object and provides calculator adapters for every engine listed above. pymatgen plays an overlapping role with a stronger materials-science bent, exposing crystallographic symmetry analysis, phase diagram tooling, and tighter integration with the Materials Project database . AiiDA layers workflow provenance on top of these libraries, recording every input, output, and code version so a multi-step calculation can be re-run or audited; Materials Cloud is the public-facing instance through which AiiDA workflows are shared with full traceability . The Materials Project itself is a curated database of more than a hundred thousand DFT-relaxed structures and derived properties, and is widely treated as the default starting point for screening candidate structures . OPTIMADE is the cross-database query API that makes Materials Project, AFLOW, NOMAD, and OQMD interoperable behind a common REST surface. In practice, a 2026 workflow rarely touches a CIF or POSCAR by hand: structures move between codes as ASE Atoms or pymatgen Structure objects, and only round-trip through file formats at the engine boundary.
The reference systems are MACE , CHGNet , M3GNet , NequIP , and Allegro ; more recent universal models include Orb and MatterSim, and the GNoME effort by DeepMind has explicitly scaled the paradigm to discovery of new stable crystals . The speed gap relative to plane-wave DFT is three to five orders of magnitude per force evaluation, which moves problems that were previously workflow-scale (a few hundred relaxations) into laptop-scale (a few thousand). The trade is in accuracy and in extrapolation: recent benchmark work shows that universal potentials systematically soften the potential energy surface, underpredicting energy barriers and high-energy configurations, but that the bias is structured enough to be partially corrected by small fine-tuning datasets [citation pending].
The shift in framing during 2024 to 2026 has been from per-system potentials trained from scratch to foundation MLIPs trained on millions of DFT calculations and then fine-tuned for the task at hand. MACE-MP, the PET-MAD family, MatterSim, and Orb are now positioned the way pretrained transformer backbones are positioned in natural language: as accurate but systematically biased priors whose remaining error can be cheaply corrected with a small in-domain dataset . The Matbench Discovery and MS25 benchmarks have become the shared reference for stability prediction and derived observables, and recent reports place Orb at roughly three to six times the throughput of earlier universal potentials at comparable accuracy . From the perspective of an MCP that prepares structures for DFT, this matters because the MLIP layer changes what relaxation means: a candidate structure can be cheaply pre-relaxed with a universal potential before any DFT call is issued, which collapses the cost of exploring tens or hundreds of geometric variations and shifts the bottleneck back onto the structure-construction step.
Workflow tooling rounds out the picture. AiiDA already mentioned for provenance is one half of the picture; the other half is the orchestration layer. FireWorks is the long-standing job manager used in Materials Project pipelines, designed for high-throughput campaigns where the same calculation template is applied to thousands of structures with retry on failure. Atomate2 is the more recent reincarnation that targets the same job shape but with a cleaner separation between workflow definition and execution backend, and ships an extensive library of preset relaxation, band-structure, and elastic-constants recipes. ASE itself ships a lighter-weight workflow surface that is more attractive for one-off scripts. The common shape across all of them is the same: a workflow is a directed graph of input-generation, simulation, and post-processing steps, with parameter sweeps fanned across the database. None of these tools, however, address the question of how a structure was constructed in the first place; they assume an input structure is already correct.
ChemCrow paired GPT-4 with eighteen expert-designed tools spanning RDKit, retrosynthesis predictors, and literature search, and demonstrated autonomous planning and execution of organic syntheses, a chromophore design loop, and a usable property-prediction interface for non-experts . ChemToolAgent extended the toolset to twenty-nine and found a nuanced result that bears directly on MCP design: tool augmentation substantially improves performance on specialized tasks involving SMILES, structure manipulation, and property prediction, but can hurt performance on general chemistry questions where the base model is already adequate [citation pending]. On the atomistic side, ChemGraph and LangSim wrap ASE as the substrate through which LLM agents can construct atomic structures and dispatch DFT or molecular-dynamics jobs, while MDAgent targets MD workflows specifically . DREAMS goes one level higher, using multi-agent coordination to plan and execute DFT screening campaigns. The MatTools benchmark uses pymatgen as a stress-test substrate and finds that even strong general-purpose LLMs frequently fail to combine its primitives correctly without external scaffolding [citation pending].
Practitioners cite a consistent set of friction points across this tooling stack. The first is initial-geometry construction. Bulk crystals are easy because pymatgen and ASE both ship reliable builders, but slabs require choosing a surface termination, a vacuum thickness, and a fixed-layer prescription; interfaces require commensurate matching of two lattice parameters across a heterojunction; and defect cells require choosing a supercell size that suppresses periodic-image interaction. Each step admits several reasonable choices that affect convergence. The second is numerical convergence: plane-wave codes require independent convergence of the kinetic-energy cutoff and the k-point sampling density, and the converged values depend on the system. Conservative defaults are wasteful, aggressive defaults are wrong, and the field has no single rule of thumb that works across surface, interface, and bulk. The third is pseudopotential consistency: mixing pseudopotentials between elements or between calculations is a documented source of irreproducible energy comparisons. The fourth is the management of variations: a real campaign produces tens to thousands of related structures (coverage scans, alloy compositions, defect species) and the bookkeeping is non-trivial without a workflow layer. The fifth is relaxation pathology: a relaxation can fail to converge, converge to a saddle point, or converge to a metastable basin, and the engine's diagnostics are designed for an expert reader. The sixth is visualization and interpretation of periodic results, where image atoms across cell boundaries and minimum-image conventions routinely confuse non-experts.
Two trends in 2025 and 2026 are changing where the friction sits. Universal MLIPs absorb most of the convergence and variation cost: when a hundred relaxations cost a minute each, the bookkeeping that AiiDA and Atomate2 were designed to amortize becomes optional, and the bottleneck moves upstream to whatever produces the candidate structures. LLM agents, meanwhile, are now technically able to call ASE and pymatgen through tool interfaces, but the comparative evidence shows that they fail on exactly the geometric reasoning steps that human practitioners cite as friction: placing an atom at a fractional coordinate, applying a symmetry operation, deciding which surface termination is physical, recognizing an image atom across a periodic boundary. The implication for an MCP that prepares structures for DFT is that it should not try to replace the engines, the workflow layer, or the MLIPs. Each of those substrates has converged and is well understood. The unaddressed layer is the conversational structure-construction surface, and that is where this report places its design effort.
The production DFT calculator landscape splits along two axes: licensing and basis-set choice. VASP remains the dominant commercial plane-wave code for solids, on grounds of speed, pseudopotential quality, and the inertia of established workflows in surface science and catalysis. Quantum ESPRESSO and ABINIT cover the same plane-wave-plus-pseudopotential niche under open licences and are the default fallback for groups that cannot or will not pay for VASP. CP2K occupies a separate corner with its mixed Gaussian-and-plane-wave (Gaussian-and-plane-wave (GPW)) formulation, well suited to large amorphous or solvated systems. GPAW pairs the projector-augmented-wave method with real-space grids or plane waves and is tightly integrated with ASE. FHI-aims is the prominent all-electron, numeric-atom-orbital code, used when pseudopotential approximations are unacceptable, for example in core-level spectroscopy or heavy-element chemistry. Cambridge Serial Total Energy Package (CASTEP) rounds out the commercial side. The choice between these codes is rarely scientific: it is set by site licence, group tradition, and whether the system needs all-electron treatment.
Around the calculators sits a thin Python layer that every practitioner touches. ASE is the de facto lingua franca for atomic structure manipulation, with a uniform Calculator interface that hides VASP, Quantum ESPRESSO, GPAW, and most other engines behind a single Python object. Pymatgen, originating in the Materials Project, is the heavier analysis library: symmetry handling via spglib, structure matchers, phase diagrams, electronic-structure post-processing, and IO for most crystallographic and engine-native formats. The two are partly overlapping and partly complementary. ASE is the live-driving layer, used to set up calculations, run relaxations, and chain steps; pymatgen is the analysis layer, used after the calculation completes. A typical workflow imports both. Neither was designed for natural-language interaction; both expose their full power through Python APIs whose error surfaces assume an interactive notebook user.
Public structure databases provide the starting-material feedstock. The Materials Project hosts roughly 150 thousand DFT-computed inorganic structures with energies, band structures, and elastic properties, accessible through a REST API and integrated tightly with pymatgen. AFLOW and OQMD play similar roles with overlapping but distinct calculation pipelines. The OPTIMADE specification provides a federated query layer across these databases and several smaller ones, so a search for a stoichiometry can be issued once and resolved against the union. For molecular chemistry the analogous resources are PubChem and the Cambridge Structural Database, with Open Babel and RDKit as the conversion glue. For periodic systems the canonical exchange format remains the CIF, with POSCAR and XYZ as engine-specific or quick-look alternatives.
High-throughput campaigns sit above the calculator and database layers. AiiDA is the most opinionated of the workflow engines, with a provenance graph that records every input, every code, and every output as immutable nodes in a database. Atomate2, the successor to Atomate, layers Materials-Project-style recipes (relax, static, bandstructure, phonon, defect) on top of jobflow and runs them through FireWorks or a similar dispatcher. ASE workflows and Custodian fill the lighter end, where the goal is to script a few dozen calculations rather than ten thousand. Common to all of these is the assumption that a human author has written the recipe and that the system's job is to execute it. None of them currently anticipates an LLM agent in the loop that may revise the recipe mid-flight.
The most consequential shift in the last three years has been the rise of machine-learning interatomic potentials. The architectural lineage runs from SchNet and ANI through Behler-Parrinello descriptors and the Gaussian Approximation Potential, then to the equivariant graph neural networks that now dominate: NequIP, Allegro, MACE, M3GNet, CHGNet, and, more recently, Orb, MatterSim, and the PET-MAD family. These models learn the potential energy surface from DFT data and evaluate it three to five orders of magnitude faster than a self-consistent-field calculation on the same structure, with energy errors that are now routinely below 50 meV per atom on Matbench Discovery and force errors competitive with the noise floor of the reference data. The decisive change is the universal model: a single set of weights covering most of the periodic table, trained on the Materials Project Trajectory and similar corpora, that can be used zero-shot or fine-tuned on a few thousand additional configurations for a target chemistry.
Matbench Discovery, the community benchmark that pits ML potentials against Perdew-Burke-Ernzerhof generalised-gradient exchange-correlation functional (PBE) DFT in a high-throughput stable-structure discovery scenario, has crystallised the current ranking. As of mid-2026, the second-generation MACE-MPA-0 model achieves a Combined Performance Score of 0.795, an energy mean absolute error of 0.028 eV per atom, and a discovery acceleration factor of 5.58 relative to a dummy baseline, trained on roughly 12 million configurations from the Materials Project Trajectory plus the sAlex dataset. Orb v2 reaches an F1 score (harmonic mean of precision and recall) (F1) score of 0.880 on the same benchmark with similar energy error. The original MACE-MP-0 and CHGNet are now mid-table, displaced by larger-data, larger-parameter successors. GNoME, the DeepMind effort, sits apart: trained on roughly 89 million configurations, it discovered 380 thousand thermodynamically stable structures below the convex hull, but is used primarily as a stability predictor rather than a molecular-dynamics force field. The practical upshot is that for the element subset of this report (Pd, Cu, C, H), a universal MLIP can produce a relaxed structure that DFT will accept as a converged starting point, at a fraction of the Self-Consistent Field (SCF) cost.
LLM agents for computational chemistry are a recent and unsettled category. ChemCrow (Bran et al., 2023) demonstrated that an LLM equipped with a suite of cheminformatics tools could plan and execute synthesis routes, retrieve molecular properties, and predict reaction outcomes; it remains the most-cited proof of concept that an LLM can coherently drive a multi-tool chemistry workflow. MD-Agent (and the closely related MDAgent of Cox et al.) ports the same pattern to molecular dynamics, with tools for input preparation, simulation launch, and trajectory analysis on top of OpenMM. Electronic-lab-notebook agents and several pymatgen-wrapping harnesses have appeared in the same window. All of these systems share a common shape: a general-purpose LLM, a hand-curated tool list, and a system prompt that nudges the model toward tool-call rather than free-text reasoning. None of them yet exposes a stable conversational interface for periodic structure manipulation at the atom level. The recently published AtomWorld benchmark quantified the gap, finding that leading LLMs fail consistently on CIF-level tasks that require symmetry operations, fractional-coordinate placement, or image-atom detection across periodic boundaries.
The friction surface that practitioners hit repeatedly is not in the calculators themselves, which are mature, but in the surrounding work. Building a sensible initial geometry is the most cited pain point: slab models require correct surface termination, vacuum padding, and dipole correction; interfaces require lattice matching across two cells; defect supercells require image-image separation that is large enough to suppress spurious interactions. A second cluster is convergence: k-point density, plane-wave or basis-set cutoff, and smearing all need per-system calibration, and the standard workflow runs a small grid of convergence tests before any production run. A third is pseudopotential consistency: mixing potentials across calculations or across codes produces silent energy offsets that are easy to miss in a downstream comparison. A fourth is managing variation count: a hydrogen-coverage study or a binding-site sweep generates tens to hundreds of related structures, and the bookkeeping is rarely scripted cleanly. A fifth is relaxation that fails to converge, oscillates between basins, or converges to a saddle point rather than a minimum; the diagnostics are buried in engine output and require an experienced reader. The final cluster is visualisation and interpretation of periodic results: charge densities, projected densities of states, and band structures are still typically inspected one calculation at a time in Visualisation for Electronic and STructural Analysis (VESTA) or matplotlib. Each of these is a place where an LLM agent with a structure-aware tool surface could plausibly remove a step that currently costs a human hour.
Cells, periodic boundaries, and image addressing
The simulation box is a unit cell when it is the smallest repeating motif of the crystal, and a supercell once it has been enlarged along one or more lattice directions to host adsorbates, defects, or low-coverage configurations. The cell is fully described by three lattice vectors a, b, c packed into a 3x3 matrix; downstream engines and structure libraries all consume this matrix as the geometric ground truth. Atoms inside the cell are addressed in one of two coordinate systems: fractional coordinates, where each component lies in the closed interval [0, 1] and is interpreted as a multiplier on the corresponding lattice vector, or Cartesian coordinates in angstroms. Both representations are exact and round-trippable through the lattice matrix, and ASE [1], pymatgen, and the major DFT engines all expose both views.</text>

The cell is repeated under periodic boundary conditions along the directions for which PBC is enabled. The replication is pure translation: image n of the cell sits at the origin shifted by n_a a + n_b b + n_c c for an integer triple (n_a, n_b, n_c). Three structural archetypes follow from this. Bulk solids enable PBC on all three axes and choose a cell that captures the relevant chemistry: a primitive cell for a clean material, a supercell to dilute an adsorbate or defect. Slabs enable PBC on two axes and insert a vacuum gap along the third, typically twelve to twenty angstroms, so that the slab does not see its own image across the gap; the in-plane lattice constants stay tied to the underlying crystal. Isolated molecules conventionally keep PBC nominally on along all axes for engine compatibility but use a cell large enough (around fifteen angstroms of vacuum in every direction) that the molecule's nearest image is non-interacting; cluster-mode codes that solve the open-boundary Poisson problem natively are the rigorous alternative.</text>

Cell-shape constraints across DFT engines are mild. Plane-wave codes including VASP, Quantum ESPRESSO, ABINIT, and CASTEP accept an arbitrary 3x3 lattice matrix; localised-basis codes such as CP2K and FHI-aims do the same. VASP requires the cell to be right-handed (the scalar triple product (a x b) . c must be positive); a left-handed cell is accepted by some preprocessing tools but produces a sign-flipped reciprocal lattice that VASP rejects at the symmetry-finding stage. The MCP enforces a positive triple product on cell write and reorders the lattice vectors if necessary to make this so, preserving the geometry while removing a class of silent failures. Standardised cell choices (Niggli reduction, the conventional cell in the Setyawan-Curtarolo basis) are not required by any engine but are useful for k-point path generation and for cross-paper comparisons; the MCP exposes them as transformations rather than imposing them on the canonical representation.</text>

Atoms, bonds, and what DFT actually consumes
A Kohn-Sham DFT calculation accepts the atomic numbers and the nuclear positions, and, for periodic systems, the cell vectors; the converged electron density is its output [3]. The bonds a chemist draws over the result, whether single, double, aromatic, or dative, are not inputs at all. They are interpretations recovered after the self-consistent solve by partitioning the density that the calculation itself produces [4]. The engine has no slot for them and does not honour them in any way during the variational minimisation. This is the single most consequential fact for an LLM-facing structure layer. The bond graph the model manipulates is a presentation contract with the LLM, not a quantum-mechanical input.
The bond graph in our schema therefore plays a strictly different role from the positions and the cell vectors. It is an ergonomics layer for the LLM. The model can ask to attach a hydrogen to aPd123 with a bond length of 1.5 angstroms, oriented sp3-style at the open coordination site, and the MCP resolves the request into a position. Once the atom is placed, the engine ignores the bond annotation. We carry the graph for three reasons. Conversational structure-building benefits from naming connectivity directly, so the model can say "the dative bond between Pd and CO" without re-deriving topology from interatomic distances on every call. Post-relax visualisation and querying are easier when topology is explicit. And comparing the input bond graph against a graph inferred from the relaxed geometry is a cheap detector for unphysical chemistry such as ring opening, bond migration or dissociation that would otherwise pass through unflagged.
The atom record we store carries a small, well-defined set of fields. The element is the atomic number Z; it fixes which pseudopotential the engine will pick up. The position is held as fractional coordinates against the cell, with a cached Cartesian for convenience. A locked flag translates to the engine's selective-dynamics input; in VASP this is per Cartesian axis rather than per atom, so each atom's T/F triple says which of x, y, z are free, and any tooling that exposes a single boolean per atom must commit to a default mapping. An optional initial magnetic moment seeds spin-polarised runs. An optional initial charge state is honoured by codes that accept one. An optional per-atom constraint slot supports uses such as constrained DFT, in which the integrated charge or spin on a fragment is fixed during the SCF cycle. Element, position and cell are the only mandatory three. Everything else is an opt-in input that the engine consumes if present and ignores otherwise.
The bond record carries an ordered pair of atom handles (i, j), an image triple attached to j defaulting to (0, 0, 0) and taking non-zero values for bonds across the periodic boundary as fixed in the addressing section , a nominal bond order chosen from single, double, triple, aromatic and dative for LLM intuition rather than for any quantum-mechanical fidelity, and a kept-during-relax flag for the rare cases where we want to expose a relax-with-constraints mode in which the engine is asked to keep an interatomic distance fixed. The bond-order field is not a Mayer or Wiberg bond index; it is a hint to the model about how to talk about the bond. If the relaxed geometry contradicts the nominal order, for example a "double" bond stretched to a single-bond distance, the contradiction is something we surface to the LLM, not something we propagate back into the engine on the next run.
Hybridisation, the familiar sp3, sp2, sp labelling of a centre, is not stored as input on the atom; it is an inferred field, recomputed from the relaxed geometry on demand. A coordination-number plus bond-angle heuristic suffices for the common cases: a carbon with four neighbours at roughly tetrahedral angles reads as sp3, three coplanar neighbours at about 120 degrees as sp2, two collinear neighbours as sp. For molecule-like fragments where this heuristic is brittle, the MCP can call out to RDKit or openbabel to perceive bond orders and hybridisation from coordinates the way a cheminformatics stack would. Storing the inferred value as an input field would let it drift out of step with the geometry. The right default is to derive it on read and to treat any stored copy as a cache that is invalidated whenever positions change.
There is no per-bond stress in DFT, and any field by that name in an output schema would be a category error. Stress is a 3x3 tensor defined on the whole simulation cell, computed from the derivative of the total energy with respect to the cell metric, and used to drive cell-shape relaxation. It does not decompose onto pairs of atoms. The defensible per-atom analogues are the Hellmann-Feynman force on each atom, which is a true output of every DFT engine, and, where the engine supports it, a per-atom potential-energy decomposition. A per-bond strain can be defined heuristically as the signed deviation of the interatomic distance from the sum of covalent radii. This is useful for surfacing a phrase like "the Pd-C bond at site k is stretched 0.3 angstroms past equilibrium" in an LLM response, but it must be flagged as a heuristic and not presented as a quantum-mechanical observable.
The atom-level locked flag is genuinely useful: fixing the bottom layers of a slab while relaxing the surface, holding a substrate atom in place during an adsorbate relaxation, pinning a reaction coordinate for a nudged-elastic-band run. But it earns its keep only when used sparingly. Heavily constrained relaxations tend to converge to high-strain, high-energy local minima rather than to physically meaningful structures. They also mask convergence problems: a locked atom has zero force on the constrained components by construction, so the engine's max-force convergence test sees an artificially small number and reports a stuck geometry as converged. The discipline the MCP should encode is to lock the minimum that the physics demands, and, where the budget allows, to compare the energy of a constrained relaxation against an unconstrained reference on the same structure.
The per-atom quantities the LLM most often wants to read, namely partial charge, oxidation state, bond order and lone-pair location, are post-SCF outputs derived by partitioning the converged density. Mulliken-style schemes project the Kohn-Sham orbitals onto the atomic-orbital basis and assign electrons by basis function; Hirshfeld and Bader schemes partition the electron density in real space, the former by reference to a free-atom promolecule and the latter by following gradient flow to atomic basins, and the two families can disagree by a sizeable fraction of an electron on the same atom [5]. DDEC6 and CM5 build on this lineage with chemically tuned reference densities. The electron localisation function is a complementary scalar field that reports the local probability of finding an electron given neighbouring electrons, and is read as a heuristic for distinguishing covalent, metallic and lone-pair regions [6]. The output measurements the MCP exposes per atom follow from this stack: the Hellmann-Feynman force vector and its magnitude, the coordination number under a covalent-radius cutoff, the list of nearest neighbours as (atom-handle, image-triple) pairs, the inferred hybridisation, a partial charge whose partitioning scheme is part of the field rather than a hidden convention, and a local strain summary computed from the bond-length deviations above. The engine sees only positions, the cell, and the optional selective-dynamics and magnetic-moment hints; the LLM gets a stable, named-field interface to the result. The split is what keeps the MCP honest about which side of the input-output boundary each concept lives on.
Storage schema and import/export formats
The structure data for a v1 system is small. A typical surface slab has fewer than two hundred atoms; the cell, fractional coordinates, bond graph, and a full set of DFT output (energy, forces, stress) for that slab fit comfortably inside a few kilobytes. At this scale the usual argument against a relational store — that it performs poorly on large binary blobs — does not apply. What does apply is the relational character of the data. A cell has many atoms; atoms participate in bonds; bonds belong to cells; calculations consume cells and produce outputs. These are foreign-key relationships that SQL handles naturally and that a file tree would require reimplementing by hand. There is a deeper structural reason to prefer Postgres in this system: the LLM interface already operates over named refs with stable handles. A cell is `, an atom is aPd123`. That handle is a primary key, and queries such as "give me all double bonds in cell 42" are a row scan, not a grep over a directory. The pattern established in precis applies here directly: refs live in the database; file formats are rendered on demand and are not the source of truth. Storing CIF or POSCAR as the primary representation would mean trading transactions, diffs, and relational queries for a scheme whose apparent simplicity evaporates once you need to update a single atom without rewriting the whole file.
Five tables cover the full structure state. The cell table holds the 3×3 lattice matrix (row vectors in angstroms), a three-element PBC flag, a free-text description, and a project tag. The atom table stores the atom handle (aPd123), the parent cell, the element symbol, fractional coordinates, a three-axis locked flag for selective dynamics, an optional initial magnetic moment, and a status field (active or removed). The bond table records a pair of atom handles, the image-offset triple of the second atom (defaulting to [0,0,0]), a bond-order enum, a constrained flag, and an inferred flag that distinguishes bonds derived from a population analysis from bonds asserted directly by the user. The calc table stores the cell reference, the engine identifier, a jsonb params field, lifecycle timestamps, the total energy in eV, per-atom forces, the 3×3 stress tensor, and a jsonb map of output artefacts. The observation table is a keyed cache: it stores the result of any expensive derived query (band structure slice, DOS, charge-density integral) indexed by (calc_id, query_hash) so repeated reads do not re-invoke the engine.
``sql
CREATE TABLE cell (
    id          serial PRIMARY KEY,
    lattice     float8[3][3] NOT NULL,
    pbc         bool[3]      NOT NULL DEFAULT '{true,true,true}',
    description text,
    project_tag text
);
CREATE TABLE atom (
    id             text PRIMARY KEY,          -- e.g. aPd123
    cell_id        int  REFERENCES cell(id),
    element        text NOT NULL,
    frac_x         float8, frac_y float8, frac_z float8,
    locked_axes    bool[3] DEFAULT '{false,false,false}',
    initial_moment float8,
    status         text CHECK (status IN ('active','removed')) DEFAULT 'active'
);
CREATE TABLE bond (
    atom_i      text REFERENCES atom(id),
    atom_j      text REFERENCES atom(id),
    image       int[3]  DEFAULT '{0,0,0}',
    order_enum  text,
    constrained bool    DEFAULT false,
    inferred    bool    DEFAULT true,
    PRIMARY KEY (atom_i, atom_j, image[1], image[2], image[3])
);
CREATE TABLE calc (
    id               serial PRIMARY KEY,
    cell_id          int REFERENCES cell(id),
    engine           text NOT NULL,
    params           jsonb,
    status           text,
    started_at       timestamptz,
    finished_at      timestamptz,
    energy_ev        float8,
    forces           jsonb,
    stress           float8[3][3],
    output_artifacts jsonb
);
CREATE TABLE observation (
    calc_id    int  REFERENCES calc(id),
    query_hash text,
    result     jsonb,
    cached_at  timestamptz DEFAULT now(),
    PRIMARY KEY (calc_id, query_hash)
);
``
This fits naturally onto the precis ref/chunk substrate: cell and atom stand in the same parent-child relationship as a draft and its chunks. The foreign-key discipline that SQL enforces for free (no orphan atoms, no calculation that references a deleted cell) would require explicit application-layer validation in a file-based scheme.
Six output formats cover the realistic interchange surface, in priority order. POSCAR and CONTCAR, the VASP geometry format, are required: every serious plane-wave code either reads them natively or accepts them via a one-step converter, and they are the de facto standard for sharing optimised geometries in the community. CIF is the crystallographic community's canonical interchange format, standardised by Hall, Allen, and Brown (1991), and is required for submission to databases such as the Cambridge Structural Database and for comparison with experimental diffraction data. Extended XYZ, a superset of the trivial Cartesian dump that encodes cell vectors and per-atom properties in the comment line, is expected by ML potential training pipelines and by visualisers such as OVITO and Avogadro; it should be available for every structure in the system regardless of whether a DFT calculation has been requested. ASE Atoms JSON, produced by atoms.write('out.json'), is the Pythonic round-trip format for handing structures to downstream ASE workflows without loss of cell or property data. A Quantum ESPRESSO &SYSTEM block plus ATOMIC_POSITIONS card is required once QE is wired as a second relaxation backend. LAMMPS data files are deferred: they are out of scope for v1 and can be added when the system gains molecular-dynamics capability.
Import is the harder direction because structure files are produced by dozens of codes with subtly incompatible conventions for units, coordinate frames, and symmetry expansion. The pragmatic answer is to delegate to ASE's IO layer. ase.io.read handles CIF, POSCAR, XYZ, PDB, Gaussian input, cube files, and many other formats through a single call that inspects the file extension and content; the returned Atoms object carries cell vectors, fractional or Cartesian coordinates, and PBC flags in a consistent internal representation [citation pending]. The import adapter for this system has a two-step interface: call ase.io.read to obtain an Atoms object, then traverse its attributes to populate the cell and atom rows. Bond inference from the imported geometry is run immediately using a covalent-radius cutoff and stored as inferred=true bonds; user-asserted bonds arrive through the MCP verb layer after the fact. This approach inherits ASE's extensive format-specific quirk handling — CIF symmetry expansion, POSCAR selective-dynamics flags, extended XYZ property parsing — rather than reimplementing it.
Reproducibility of DFT results requires storing not just the output numbers but the exact computational context that produced them. The calc table therefore carries: the engine name and version string; the pseudopotential set for each element with a SHA-256 checksum of each pseudopotential file; the k-point mesh specification (Monkhorst-Pack grid dimensions or a list of explicit k-points); the plane-wave kinetic-energy cutoff in eV; the smearing type and smearing width; and the exchange-correlation functional identifier. These fields are non-negotiable. Energy differences between calculations that used different pseudopotentials or different cutoffs are physically meaningless even when the numbers are numerically close, and the FAIR data provenance discipline formalised in high-throughput DFT infrastructure — including the TribChem FAIR workflow and the AiiDA and Fireworks lineages — treats these fields as metadata that must travel with every result, not as commentary that can be reconstructed later [citation pending]. The output_artifacts field stores paths to the engine stdout, the charge-density file, and any wavefunction restart, keyed by a stable convention-name so that the observation cache can locate them without parsing filenames.
Atoms are identified by handles of the form a<Element><n>, where the integer n is a project-global counter that increments monotonically. An atom added to any cell in the project receives the next available integer, so aPd1 through aPd17 may belong to three different cells; the global scope prevents aliasing — when the LLM refers to aPd7, that handle is unambiguous within the project regardless of which cell the conversation is currently about. Deletes are soft: setting atom.status = 'removed' excludes the atom from active queries and from engine export, but the row and its primary key remain in the database. This is the same invariant that makes precis chunk handles stable: once minted, a handle is never recycled, so references in calculation logs and bond records that point to aPd7 do not become stale when the structure is edited. The consequence for the MCP verb layer is that the remove_atom verb writes a status update, not a SQL DELETE, and that the LLM can address a removed atom in queries that explicitly request structure history.
Relaxation engine selection
DFT relaxation engines are forty years of accumulated code-level optimisation, and the variational, force-evaluation, optimiser, and pseudopotential machinery inside a production engine sits at the boundary of several active research fields. Reimplementing this stack inside the MCP would be a multi-PhD undertaking that would produce something slower and less accurate than VASP or Quantum ESPRESSO. The MCP wraps. Its system.relax verb exposes a single signature, system.relax(method='cheap'|'medium'|'dft', tol=..., max_steps=...), and dispatches to a tiered ladder of engines. The tier is the LLM's only knob on the cost-accuracy axis. Everything below it (basis, pseudopotentials, k-mesh, optimiser, smearing) is set by the MCP from validated defaults per tier and is recorded in the structured result so the answer is reproducible.
The cheap tier covers the workload the LLM creates for itself: an atom has just been inserted at a guessed position, two fragments have just been combined, or a bond has just been substituted. Gross strain on these structures is dominated by close contacts and bad angles that any reasonable force model can drain in milliseconds. The right machinery is a foundation machine-learning interatomic potential trained across the periodic table: MACE-MP-0 [7], M3GNet [8], CHGNet, and Orb. M3GNet was trained on the full Materials Project relaxation database with structural relaxation as the named target application [8], and the published reference pipeline composes M3GNet with the Fast Inertial Relaxation Engine (FIRE) optimiser inside ASE for the actual minimisation [8]. On the Matbench Discovery leaderboard, foundation MLIPs trained on the current generation of large datasets reach F1 scores above 0.9 for thermodynamic stability and around 20 meV/atom accuracy on formation energy [9], comfortably better than the threshold needed to detect the unphysical geometries the cheap tier is meant to reject. The MCP routes a pre-relax through the cheap tier before every DFT call and refuses to spend DFT budget on a structure whose ML pre-relax produced an unphysical bond length or an implausible per-atom energy.
The medium tier is semi-empirical: GFN2-xTB and the Density Functional Tight Binding (DFTB)+ family. GFN2-xTB is a self-consistent tight-binding method with multipole electrostatics and density-dependent dispersion contributions, and it runs at seconds per single-point on systems of order one hundred atoms. The fit to v1 chemistry is partial. GFN2-xTB and the published DFTB Slater-Koster parameter sets handle main-group organics and many small clusters well, but their accuracy on transition-metal surfaces is uneven, and the existing parameter sets do not always cover the specific Pd or Cu environment of interest. Since v1 is built around Pd and Cu surfaces with C and H adsorbates, the MCP should make this conditional honest: route to the medium tier when the system is dominated by C and H, and skip the medium tier (going straight to DFT) when Pd or Cu chemistry is structural rather than incidental.
The DFT tier wraps the production codes that already exist: VASP, Quantum ESPRESSO, CP2K, and GPAW. The connecting glue is ASE [1], whose Atoms object plus Calculator protocol gives a uniform Python contract (atoms.get_potential_energy(), atoms.get_forces(), atoms.get_stress()) across every one of these engines [10]. The MCP relax verb only ever needs to know about that interface. The engine binary, its input file format, its k-point specification, its pseudopotential set, and its output parsing all live in the calculator implementation that the engine community already maintains. This is the correct boundary because each DFT code already invests heavily in its own ASE binding, and the MCP inherits that investment by adopting ASE rather than re-wrapping each engine itself.
The force engine and the optimiser are two separate choices. ASE and pymatgen both ship the standard optimisers: Broyden-Fletcher-Goldfarb-Shanno quasi-Newton optimizer (and its limited-memory variant L-BFGS) (BFGS), L-BFGS, FIRE, and MDMin. Any of them can be paired with any calculator. The convention that has settled in the field is L-BFGS for DFT and FIRE for the ML and semi-empirical tiers [11][8]: L-BFGS converges in fewer ionic steps when each force evaluation is expensive, while FIRE is more robust against the noisier potential energy surfaces and the larger initial strains the cheap tier sees. Cell relaxation is a further axis. ASE exposes it through ExpCellFilter or FrechetCellFilter, which wrap the Atoms object so the optimiser sees a combined vector of atomic and strain coordinates; VASP exposes it directly through ISIF=3. These are not the same algorithm. They parameterise the strain differently and apply different defaults to the cell-pressure penalty, and the MCP must record which one was used in the relax result so that two structures coming back from the same verb can be compared honestly.
The convergence contract is unambiguous: a force tolerance in the 0.01 to 0.05 eV/Å range is the field-standard target for production geometries, and a max-step ceiling (commonly one hundred) bounds runaway. Silent failure is the dangerous mode. If the optimiser plateaus before tolerance, the verb must return converged=false with the trajectory and the offending residual force, not raise an exception and not pretend the last frame is the answer. The MCP relax result is therefore a structured value with a fixed shape: a converged boolean, the final total energy, the maximum and RMS forces, the number of ionic steps, the optimiser name, the cell-filter choice if one was used, and a list of trajectory chunk handles for the LLM to inspect. The same shape applies across all three tiers, so the LLM never has to learn three response languages.
What the MCP does not do is reimplement the inside of an engine. Solving the self-consistent Kohn-Sham equations, integrating over the Brillouin zone with a converged k-point mesh, choosing and validating pseudopotentials, handling Pulay stress when the plane-wave basis changes shape with the cell, and selecting a smearing function and width whose energy is meaningful for a metallic system are each a research field with its own thirty-year correction history. Even the MACE-MP-0 paper frames its own stable MD propagation across the periodic table as a landmark only after a decade of accumulated training data [7]. We are not going to beat VASP at metal smearing in six months, and we are not going to beat Quantum ESPRESSO at pseudopotential consistency. What remains is a clean MCP-side workload: orchestration (which tier to call and when), caching (the same relaxed Pd(111)+H slab should not be recomputed every time the LLM asks), retry logic (an SCF failure on a tight tolerance should auto-resubmit at relaxed settings before bothering the LLM), ML pre-relax (above), structure validation before and after (no overlapping atoms, no broken cells, the bond graph predicted from the relaxed geometry agrees with the input within a reasonable tolerance), and error messages that an LLM can act on (for example, "Pd-H distance 0.7 Å after preview relax is unphysical; the canonical Pd-H bond is roughly 1.55 Å, try inserting H at the canonical hollow or bridge site instead"). A final cheap discipline is gradient-of-force consistency: ASE's atoms.get_forces() and atoms.get_stress() are the contract, and the calculator is trusted, but the MCP can warn at no cost when force times displacement does not predict the energy change between optimisation steps. That mismatch is a classic signature of a non-self-consistent SCF or an under-converged grid, and surfacing it as a structured warning saves the LLM from chasing the symptom downstream.
MCP interface: verbs, addressing, and observation
The verb surface follows the precis discipline: small, composable, deterministic. Reading is the verb \texttt{get}; mutation is split across \texttt{put}, \texttt{edit}, \texttt{tag}, \texttt{link}, and \texttt{delete}; observation is just a computed \texttt{get} over the same address space. Every mutating call returns the new state of the touched refs and the version of the cell they live in, so the model never has to guess what its own action did. Every read is idempotent and addressable, so the model can repeat any observation without paying the cost of recomputing it. The interface borrows the unified client-server contract that [12] introduces for MCP servers in general, and grafts onto it the object-oriented Atoms / Calculator separation that ASE has made canonical for Python atomistic work [13].
Addressing is the load-bearing design choice. Every atom is minted with a stable global handle of the form \texttt{a<Element><int>}, so a palladium atom might be \texttt{aPd123} and a hydrogen \texttt{aH98}. The element prefix is a redundancy the LLM finds useful: when it sees \texttt{aPd123} in a tool result, it does not need a follow-up call to remember what species that is. Cells carry slug handles such as \texttt{}, so a structure is named the same way a document is. Bonds are not addressed directly. A bond is a derived relation between two endpoints, and the LLM addresses it through its endpoints (\texttt{measure.distance(aPd123, aH98)}); this avoids the bookkeeping nightmare of keeping bond ids in sync with structural edits, and it mirrors the way pymatgen treats neighbour relations as queries over a Structure rather than as first-class objects [14].
Periodic images attach to any handle via a trailing bracket triple: \texttt{aPd123[-1,1,1]} is the periodic image of \texttt{aPd123} shifted by minus one lattice vector along \(\vec{a}\), plus one along \(\vec{b}\), plus one along \(\vec{c}\). A frequent worry is whether to disallow corner crossings such as \texttt{[1,1,1]}, on the grounds that simultaneous shifts along all three axes are physically rare and usually point at a bug. The recommendation here is to permit them. The image triple is an addressing primitive, and addressing primitives should be total. The job of distinguishing the physically reasonable image from the pathological one belongs to the query layer, not the address space. The minimum-image rule lives there, exposed as \texttt{view.neighbors(of=...)} or \texttt{nearest\_image\_of(i,j)}: a query returns the image with the smallest distance, and the LLM picks up the right \texttt{[dx,dy,dz]} as part of the result. Restricting the addressing grammar would force the LLM to special-case every PBC edit it tries to express, and would leave it no way to talk about a corner image when one is actually wanted (a long thin nanowire viewed along its diagonal, for example).
The verb table below is the canonical surface. It is small on purpose. Every verb maps to a single conceptual action; combinations of verbs build up everything else. The naming follows the ASE \texttt{Atoms} / \texttt{Calculator} split [13] (geometry-only verbs do not invoke a Calculator, and energy-bearing verbs do) and the pymatgen \texttt{Structure.get\_neighbors} convention [14] for queries that return ranked lists.
\begin{verbatim}
cell.set(lattice=..., pbc=...)                          # define or replace the cell
atom.insert(element, frac=(x,y,z) | cart=(x,y,z),       # returns aXxN handle
            locked=[True/False]*3)
atom.insert_pattern(template='sheet'|'line'|'cube'      # bulk insert; see "templates"
                   |'cluster', element, **params)
atom.move(id, frac | cart)
atom.lock(id, axes=[True/False]*3)
atom.delete(id)
bond.add(i, j, image=[0,0,0], order=1, constrained=False)
bond.remove(i, j)
bond.constrain(i, j, length=...)
system.relax(method='cheap'|'medium'|'dft',             # returns a relax_result handle
             fmax=0.05, steps=200, cell=False|True)
view.slice(plane=(point, normal), thickness=...,        # CAD-style observation
           format='ascii'|'png')
view.neighbors(of=aPd123, within=3.0)                   # -> [(j, image, distance), ...]
measure.distance(i, j[, image])
measure.angle(i, j, k)
measure.dihedral(i, j, k, l)
measure.coord_number(of=aPd123, cutoff=...)
measure.hybridization(of=aPd123)                        # inferred from geometry
measure.charge(of=aPd123, method='bader'|'mulliken'|'ddec6')
query.formation_energy(reference=...)
query.dos(...)
query.bandgap(...)                                      # valid only after a DFT calc
export.format(format='poscar'|'cif'|'xyz'|...)          # returns a workspace file handle
\end{verbatim}
The CAD-style inspection ray translates uncomfortably to atomic systems. A ray cast through a CAD model returns a sequence of surface intersections, which is meaningful because the underlying object is a manifold. An atomic system is a sparse point cloud, and a literal ray through it almost always misses every atom. Two ergonomic primitives are recommended in its place. \texttt{view.slice} returns a 2D projection of every atom that falls within a slab of specified thickness centred on a plane; the projection can be rendered as ASCII for quick LLM sanity checks or as a PNG for human review. \texttt{view.neighbors} returns a structured neighbour list, with periodic-image triples and distances, and is the verb the LLM actually wants most of the time. The LLM reasons over named atoms, not pixels, and a neighbour list is the form in which that reasoning is cheapest. The two together cover the two questions a CAD inspection ray normally answers: "what does the structure look like through this plane" and "what is near this atom".
Every mutating call bumps a per-cell version counter. Observations cache by the tuple (cell\_version, query\_hash), so the second call to \texttt{measure.coord\_number(of=aPd123, cutoff=2.8)} returns the cached integer in microseconds, and the third call after a single \texttt{atom.move} recomputes only because the version has changed. This lets the model interrogate the same structure repeatedly without paying compute twice, and it lets it diff a structure against a previous version by comparing observation tuples. The version counter doubles as a lightweight undo handle: the LLM can ask for the cell at version \(n\) and recover the geometry that produced a previous result, which is useful when a relaxation diverges and the model wants to retry from the last known-good state.
The error envelope is designed for an LLM client. Every verb returns either a success envelope \texttt{\{ok, ref, warnings\}} or a failure envelope \texttt{\{error: \{code, message, suggested\_fix\}\}}. The \texttt{suggested\_fix} field is the part the LLM benefits from most. A bare error string ("Pd-H distance below covalent-radii sum") forces the model to reason about what to do; a structured suggested fix ("Pd-H distance 0.6 \r{A} is below the covalent-radii sum of 1.85 \r{A}. Did you mean fractional 0.5 0.5 0.5 rather than 0.05 0.05 0.05?") often resolves the call in one extra tool turn. Warnings are kept distinct from errors and never block. A warning indicates "the call succeeded but you may not have meant it": inserting an atom outside the cell, creating a bond longer than the covalent-radii sum by more than a factor of two, requesting a relax with no calculator pinned. The model can choose to ignore warnings, but it sees them in the same envelope as the success ref, so they enter its planning loop without an extra call.
The \texttt{atom.insert\_pattern} templates compress what would otherwise be dozens of \texttt{atom.insert} calls into one verb, and they align with how a domain expert verbally describes a structure. A \texttt{sheet} template takes a Miller-indexed plane (hkl), a number of atomic layers, two in-plane lattice vectors, and an element, and inserts a 2D lattice of that element along the plane. A \texttt{line} template takes a direction vector, a pitch, and a length, and inserts a chain. A \texttt{cube} template takes an axis-aligned bounding box, a lattice mode (simple cubic, fcc, bcc, hcp), and a lattice constant, and fills the volume with the corresponding lattice. A \texttt{cluster} template takes a center, a radius, and a lattice mode, and fills a sphere. The atoms minted by a pattern call all carry standard \texttt{aXxN} handles, so downstream verbs treat them identically to atoms minted one at a time; the template is purely an authoring shorthand. The choice of templates is informed by what the corpus shows researchers actually building: passivation layers, adatom chains, metallic slabs, and clusters dominate the structural prep of the papers held here.
The verbs above compose with the rest of the precis substrate without extra plumbing. Every \texttt{system.relax} call produces a \texttt{calc} ref carrying the converged geometry, the final energy, the force tail, and the method tag; that ref can be cited from a downstream argument exactly the way a paper chunk can. Every observation \texttt{get} returns a ref the next tick can read without recomputing. Every structural lineage (a sheet derived from a slab, a defect derived from a pristine cell) is recorded with a \texttt{link(rel='derived-from')} edge, so the provenance of any structure is a single graph walk away. The model can therefore plan a multi-step study (build, relax, perturb, relax, compare) and have the entire trail be addressable, citable, and reproducible, without ever leaving the verb surface defined here. The interface is small; the substrate underneath does the bookkeeping.
Discussion
LLM-specific challenges and ergonomic implications
The first thing the API must accept about its caller is that current LLMs reason poorly about three-dimensional coordinates and even more poorly about wraparound under periodic boundary conditions. Recent work that tries to make an LLM produce Metal-Organic Framework (MOF) crystal structures has to invest in a three-stage curriculum (spatial-aware continued pretraining, structural supervised fine-tuning, and matching-driven reinforcement learning) just to lift the model to the point where block-level assembly becomes reliable [15]. The practical conclusion for a DFT MCP is that the model should be steered out of raw coordinate space whenever it can be. Operations like view.neighbors(of=aPd123), insert_above(site=hollow_fcc), or replace(atom=aH7, with='OH') let the LLM work in a named-neighbor space the surface chemistry community already thinks in, and the server resolves these names into Cartesian deltas. Coordinates remain available, but they become the floor of last resort rather than the default surface.
Convergence is the second LLM-shaped pitfall. A non-converged DFT relax returns a number, and a model that only sees that number cannot tell whether the geometry it received is a useful answer or numerical garbage. The benchmark literature on chemistry-facing LLMs already flags overconfident predictions as a recurring failure mode [16], and a relax that quietly stops at the step limit is exactly the kind of input that turns confidence into a liability. The MCP response schema therefore has to make the convergence contract explicit. Every relax result must carry a converged boolean, the max-force at exit, the step count, the optimizer's energy and force trajectory in summary form, and a final-state hash. The error envelope must distinguish at least three failure modes: did not converge within the step budget, converged to a structure whose energy is an outlier relative to the reference set, and converged with large residual forces on atoms that were supposed to be frozen. Each of these is a different repair, and collapsing them into a single status code forces the LLM to guess.
Hallucinated chemistry is the third LLM-shaped pitfall, and it is well documented at the property-prediction level: LLMs confidently generate facts that contradict basic domain knowledge when their training data thins out [17]. In structure space the same failure looks like a confidently proposed Pd-H bond at 0.6 angstrom, a five-coordinate carbon in a saturated environment, or a hydroxyl placed inside another atom's covalent shell. The right place to catch this is not the SCF, which will diverge expensively or converge to a meaningless minimum, but a thin validation layer between the LLM's mutation request and the queued calculation. Every proposed bond should be checked against the covalent-radii sum and the typical bond-length distribution for that pair; every coordination environment should be checked against the element's valence range. A rejection should return a structured error that names the offending atoms by handle, gives the measured and expected distances, and suggests a concrete fix such as relax the new fragment with a cheap potential before reusing it. The validation is microseconds; the avoided DFT call is hours.
A cheap pre-pass with a universal machine-learning potential is the natural extension of that validation layer. Foundation MLIPs such as MACE-MP-0 [7] and CHGNet [18] now relax arbitrary inorganic systems out of the box, at a cost roughly four to six orders of magnitude below a plane-wave SCF on the same cell. Every LLM-proposed mutation should be MACE- or CHGNet-relaxed before being offered as DFT input. The pre-pass acts as a chemistry-sanity filter that does not require an explicit bond-distance rule: if the structure barely budges and the potential energy remains finite, the proposed chemistry is at least topologically plausible; if atoms drift by several angstroms or the potential energy diverges, the LLM proposed something the universal model cannot make sense of and the DFT call should be refused with that diagnostic attached. The pre-pass also gives the MCP a free initial guess for the DFT optimizer, which on its own typically halves the SCF step count to convergence.
Templates beat hand-placed atoms in practice. When an LLM is asked to construct a 64-atom Pd(111) slab by emitting a CIF, the failure mode is not subtle: lattice vectors disagree with the atomic basis, surface terminations are inconsistent, and the vacuum layer is rarely the size the model claims it is. The same model, asked to call insert_slab(element='Pd', surface=(1,1,1), layers=4, vacuum=15) on the MCP, gets it right the first time, because the server owns the geometry and the LLM only owns the intent. The same gap reappears across the template families this report has already enumerated (sheet, line, cube, slab, cluster, adsorbate): templates carry constraints that the LLM does not have to learn or remember, and they keep the model in the named-handle regime rather than dropping it into raw coordinates. This is not an ergonomic flourish. For the current model generation it is the difference between a workflow that produces correct calculations and one that does not.
Provenance and reproducibility are first-class outputs, not exhaust. A typical Python notebook session that runs fifty relaxations leaves behind a folder of pickled atoms objects and a notebook nobody else can rerun. An LLM session that runs fifty relaxations through this MCP leaves behind a graph of typed refs: each calc node carries its input geometry, the exchange-correlation functional, the k-mesh, the planewave cutoff, and the resulting energies and forces; each derived-from link points back to the structure that seeded the next one; each scientific claim the LLM eventually writes is anchored to a chunk-level citation. Tomorrow's LLM can read that graph and reconstruct what yesterday's LLM was doing without rerunning a single SCF, and a human reviewer can audit the chain end to end. The discipline is cheap on the server side and impossible to add after the fact, which is why it must be wired into the kind taxonomy rather than offered as an optional logging mode.
Some decisions remain genuinely human. The choice of a 4x4x1 versus a 6x6x1 Monkhorst-Pack mesh, the decision to enable spin polarization for a putatively non-magnetic surface, the upgrade from PBE to Heyd-Scuseria-Ernzerhof screened hybrid exchange-correlation functional (2006 parameterisation) (HSE06) for a defect level near the band edge: these are value judgments that trade cost against accuracy in ways the LLM does not have the cost budget to evaluate alone. The MCP should surface them as structured asks rather than free-text prompts. A well-formed ask carries the question, the available options, the cost multiplier of each option relative to the current plan, the expected accuracy delta, and a default the server is willing to take if the human does not reply within a deadline. Routing those asks through the ask-user tag means the agent yields cleanly, the supervising human sees a queue of decisions instead of an interrupt, and the audit log records both the question and the answer for later inspection.
It is worth being explicit about what the LLM does not need, because the temptation to build a graphical workbench is real and would waste the budget. The LLM does not need real-time visualisation: a rendered PNG slice or a top-down projection of the final configuration is enough to ground its next decision, and the cost of a one-shot render is negligible against the SCF. It does not need continuous animation of a relax trajectory: the converged endpoint and a force trajectory summary carry the relevant information, and watching the optimiser walk is a human pastime. It does not need mouse-style picking: named handles (aPd123, , ) are strictly better for an agent that produces text and consumes text, and they survive serialisation in a way that pixel coordinates do not. The MCP described here is a structured-data interface for a non-visual reasoner, not a GUI for a chemist, and conflating the two has historically produced tools that serve neither audience well.
Coordinate-level geometric reasoning is a known weakness of current LLMs. A model that confidently plans a multi-step retrosynthesis still miscounts the oxygens around a tetrahedral site, and that error rate grows once PBC are added: an atom at fractional coordinate (0.97, 0.02, 0.50) sits next to an atom at (0.03, 0.98, 0.50), but only if the model first remembers to wrap. The design lets the LLM avoid raw coordinates wherever a chemistry-level concept will do. The view layer exposes named handles and named-neighbour queries (view.neighbors(of=aPd123, within=3.0Å)), so questions like "which oxygens coordinate this metal?" become discrete graph lookups rather than 3D arithmetic. The handles wrap their own minimum-image bookkeeping, so the LLM never has to think about periodic wraparound itself.
A DFT energy returned in isolation is the most dangerous output the calculator can produce, because a non-converged relax is numerically indistinguishable from a converged one. Every relax response therefore carries an explicit envelope: an unambiguous converged boolean, the maximum residual force in eV/Å, the number of optimizer steps taken (and the cap), the optimizer family (FIRE, BFGS, L-BFGS), and a compact trajectory summary giving the energy and the max force at each step. The envelope further distinguishes "did not converge within the step budget" from "converged to a suspicious minimum"; the latter is flagged when the final energy is an outlier against a held reference, when a locked atom still carries a large residual force, or when an MLIP single-point on the converged geometry disagrees with the DFT energy by more than a model-set tolerance. The LLM is told which kind of failure it has, not just that the number is small.
LLMs will confidently propose a Pd-H bond at 0.6 Å, a five-coordinate carbon, or two atoms occupying the same site. Hallucinated-but-syntactically-valid outputs are a documented failure mode of LLMs in cheminformatics workflows [19], and the cheapest response is a structured validator that refuses the input before any compute is spent. Each refusal carries the rule it violated (bond length below the sum of covalent radii by N%, valence above the typical maximum for that element, atomic overlap below a hard-sphere cutoff), the offending measured value, and a suggested fix expressed in the same template vocabulary the LLM uses to propose moves in the first place. A check that takes microseconds blocks a prompt-to-job failure mode that would otherwise burn minutes of GPU time per mistake — and burn it on a calculation guaranteed not to converge.
Every LLM-proposed structural mutation should be relaxed with a cheap MLIP before it is allowed near DFT. MACE-MP-0 and its sibling foundation potentials are transferable across most of the periodic table and run thousands of times faster than DFT on the same cell [7]. Used as a guardrail rather than as the calculator of record, the MLIP answers a single binary question: under a model that gets covalent chemistry approximately right at near-zero cost, does the proposed structure barely budge (plausible) or does it fly apart (the LLM proposed nonsense)? The cost is negligible against any subsequent SCF, and the false-negative rate is tolerable for a screening layer. A structure that passes the MLIP check might still be a bad DFT input, but a structure that fails it almost never makes the DFT calculation worth running.
The same LLM that builds a wrong 64-atom slab atom-by-atom builds a correct one when it instead says "insert an fcc Pd slab, 4 layers, (111) surface, 3×3 in-plane". The template library — sheet, line, cube, slab, cluster, adsorbate — is not syntactic sugar over a coordinate API; empirically, it is the difference between a usable workflow and one that quietly produces broken inputs. The templates encode the regularities the LLM already knows about (close-packed planes, Miller indices, named adsorption sites) and hide the regularities it gets wrong (lattice vectors, fractional coordinates, periodic-image bookkeeping). The same principle extends to mutation: rotate-about-bond, swap-element, scale-cell, and named-site adsorbate placement are levers the LLM can pull correctly because each verb names what the operator means, not where the atoms end up.
An LLM session that does fifty relaxations should leave behind a trail that a different LLM can read tomorrow and pick up from. Every calc ref is content-addressed and cached, so re-asking the same SCF returns the same answer for free; every structure carries derived-from links back to whatever produced it, so the chain "bulk Pd → (111) slab → 3×3 supercell → CO adsorbed at hollow site" can be walked in either direction; every quantitative claim the LLM commits to the report carries a citation ref pinned to the chunk that supports it. By contrast a typical Python notebook session leaves behind a sequence of cells whose execution order has decayed, intermediate variables that were never saved, and a relationship between input and output that has to be guessed. The MCP turns reproducibility into structured state by default, rather than a discipline the operator has to remember.
A subset of decisions are genuine value judgements — which k-mesh to converge to, whether a spin-polarised calculation is needed for this transition-metal centre, whether the chemistry of interest justifies switching from PBE to HSE06 at thirty-fold cost — and these belong on the operator, not on the LLM. The MCP makes those asks structured rather than free-text. The ask carries the question, the candidates with their cost estimates ("HSE06 will multiply this job's wall time by roughly 30×; proceed?"), and the default the LLM would otherwise take. Recurrent asks of the same shape ("the operator always picks PBE for screening and HSE06 only for the final number on a publication candidate") become candidates for promotion into project-level defaults rather than per-job decisions.
The MCP deliberately does not provide what a graphical client would. There is no realtime visualisation loop; a PNG slice on demand is sufficient for the LLM to confirm "yes, this is a 4-layer slab with the adsorbate where I asked for it". There is no continuous animation of an optimisation trajectory; the converged endpoint plus the per-step summary is what survives into the report. There is no mouse-style picking; named handles, named-neighbour queries, and template verbs are strictly better than coordinate-pixel interaction for an agent that is going to be asked to repeat the workflow tomorrow. The deliverable is an API designed for an LLM operator and for the human reader of the resulting report — explicitly not a port of an existing GUI.
Conclusion
The bottleneck in modern DFT practice is no longer the physics engine, the workflow manager, or the cost of a single relaxation. The DFT codes are mature, AiiDA and Atomate2 handle bookkeeping, and the 2025 to 2026 generation of universal MLIPs (MACE-MP, PET-MAD, MatterSim, Orb) absorbs the bulk of relaxations at minute-scale cost . What remains stubbornly manual is the construction surface: building the cell, placing the atoms, choosing the surface termination, getting the periodic images right, and iterating on a candidate geometry. Existing LLM agents call ASE through tool wrappers, but the comparative literature is consistent that they fail on exactly the geometric reasoning steps practitioners cite as friction [dc1571450, dc1572481]. The MCP designed in this report occupies that layer: a small, deterministic verb set that lets an LLM drive structure construction the way it drives a code editor, with all physics deferred to established engines.
The recommendations crystallise into five load-bearing choices. Structures live in PostgreSQL as cells, atoms, and bonds, addressed by stable global handles of the form a<Element><n> such as aPd123; the element prefix is redundancy the LLM finds useful and the global counter prevents aliasing across cells [dc1571458, dc1572264]. Periodic boundary conditions are made explicit through image triples [i,j,k] carried on bonds and on cross-cell queries, so the model never has to infer wraparound from coordinates . The verb surface is small on purpose: insertion primitives at the atom, line, sheet, and cube level; local mutations (move, lock, remove, set_bond); a single relax call; and inspection verbs that return structured neighbour, distance, and bond-graph data rather than free text . Wherever possible the server steers the model out of raw Cartesian coordinates and into named-site operations such as insert_above(site=hollow_fcc) and replace(atom=aH7, with='OH'), which resolve to coordinate deltas internally and directly address the documented LLM weakness on three-dimensional geometry [dc1572481, dc1572486]. Relaxation is delegated, not reimplemented: ASE provides the calculator contract, an MLIP performs the cheap pre-relax, and a DFT engine finishes only when the structure has cleared the cheap filter [dc1571459, dc1572449].
Around those choices sits a clear stance on what the model decides and what the operator decides. Geometric construction, neighbour walking, template instantiation, and routine convergence checks belong to the model; functional choice, k-mesh convergence, spin-polarisation, and the decision to spend a thirty-fold cost increment for HSE06 on a publication-candidate calculation belong to the operator, surfaced as structured asks rather than free-text questions . The Postgres substrate is what makes both sides workable. Every cell, atom, bond, relaxation, and ask is an addressable, queryable, versioned record ; failures are observable rather than hidden inside a script the model wrote and discarded; and the same structure can be exported to CIF, POSCAR, or a calculator-specific input on demand [dc1571458, dc1572246]. The MCP is not a new physics layer. It is the construction-and-bookkeeping layer that the existing physics layers never had.
The first slice should be deliberately narrow: the Pd, Cu, C, and H element set; cubic and orthorhombic cells; MOF-flavoured and small-molecule-on-metal workloads; an MLIP pre-relax on every geometry with optional DFT finishing through ASE's calculator contract. That slice is small enough to validate end-to-end and large enough to be useful for catalyst-screening and adsorbate-placement workflows the literature already documents [dc1571450, dc1571459]. From there, the natural extensions follow without redesign: more elements as the bond-length and coordination tables grow, hybrid functionals as cost budgets allow, integration with AiiDA or Atomate2 as a downstream workflow step, and richer named-site vocabularies as templates accumulate. The validation that matters is not internal benchmarks but adoption: whether a practitioner who today edits POSCAR files by hand finds it faster, less error-prone, and more reproducible to drive an LLM that drives this MCP. If that user shows up, the design is correct. If not, the verb surface and the named-site vocabulary are where to look first.

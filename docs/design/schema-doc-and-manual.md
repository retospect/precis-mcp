# Generated schema diagram + system manual — design plan

- **Status**: proposed (2026-06-19); implementation in the same branch.
- **Authors**: Reto + agent
- **Motivation**: the hand-drawn `schema-v2.puml` drifts (it still
  shows the dropped `ref_segments` tables and misses `parent_id` /
  `prio`). A diagram **generated from the live database** can't drift.
  Separately, there is no single system overview — knowledge is spread
  across 80+ docs with no front door. Add a generated schema and a
  thin, link-heavy manual, both Markdown-on-GitHub (no PDF, no build
  step — GitHub renders Markdown + Mermaid inline).

## 1. `precis schema-doc`

A new CLI subcommand that introspects `pg_catalog` /
`information_schema` and emits a **Mermaid `erDiagram`** wrapped in a
generated Markdown file (`docs/design/schema.md`).

- **Source of rows — two transports, one renderer:**
  - `--dsn <DSN>` (default `PRECIS_DATABASE_URL` via `resolve_dsn`):
    connect with psycopg and run the introspection query. This is the
    path that works **inside the container / on a cluster node**, where
    a full-credential DSN is in the env.
  - `--from-tsv <PATH|->`: read pre-fetched introspection rows
    (tab-separated) instead of connecting. This is what the prod
    wrapper uses (see below), because a dev host can't authenticate to
    prod's pgbouncer directly — the password lives in caspar's
    `.pgpass`.
- **Renderer is pure.** `render_mermaid(cols, fks, *, snapshot)`
  takes parsed rows and returns the full Markdown string. No DB, no
  I/O — unit-testable on fixture rows.
- **Output:** `-o <PATH>` (default `docs/design/schema.md`; `-` =
  stdout). The file carries a `DO NOT EDIT — generated by
  precis schema-doc` banner + the snapshot label.
- **Scope:** base tables only (views `v_*` filtered out). One
  `erDiagram` with every table + its columns (PK/FK marked) and one
  relationship edge per foreign key, plus a compact Markdown table
  listing (name + column count) so the doc is useful even where
  Mermaid doesn't render.

### The introspection query

One `UNION ALL` stream, tab-separated, discriminated by a leading
`col` / `fk` field, so a single psql invocation (or one psycopg query)
produces everything the renderer needs:

- `col` rows: `table, ordinal, column, data_type, PK?, FK?`
- `fk` rows: `from_table, _, from_col, to_table, to_col`

Type strings with spaces (`timestamp with time zone`) are sanitised to
Mermaid-safe tokens (`timestamp_with_time_zone`) since Mermaid
attribute types must be a single token.

## 2. `scripts/gen-schema` — the baked prod path

Prod is reachable only as `ssh caspar 'psql …'` (caspar's `.pgpass`
holds the `agent_rw` password; a tunnel from a dev host hits
`fe_sendauth: no password supplied`). So the regen wrapper runs the
introspection **on caspar** and renders locally:

```sh
ssh caspar "psql -h <prod> -U agent_rw -d precis_prod -tA -F$'\t'" \
    < introspect.sql \
  | uv run precis schema-doc --from-tsv - --snapshot "precis_prod @ <date>" \
      -o docs/design/schema.md
```

The wrapper is the "always hits prod" entry point; the CLI stays
generic (any DSN / any TSV) and testable.

## 3. `docs/architecture.md` — the manual

A single narrative system overview, **Markdown on GitHub, not PDF**
(diffable in PRs, renders Mermaid inline, no build pipeline, fits the
"update in the same commit" discipline). It is deliberately
**thin and link-heavy** — it ties the pieces together and points at
the authoritative layer for each, rather than restating it (which
would just become doc #N+1 that drifts):

- the seven-verb surface + `kind=` dispatch → `precis-overview`
- the kinds catalogue → `precis-overview` / `precis-help`
- storage → the generated `schema.md` (embedded) + `storage-v2.md`
- the todo-tree / jobs / review tiers → `CLAUDE.md` + the design docs
- workers + discovery (F20) → `CLAUDE.md`
- decisions → `docs/decisions/README.md`

## 4. `docs/README.md` — the landing index

The `docs/` tree (design / decisions / user-facing / conventions, 80+
files) has no front door. Add a one-screen index: what each
subdirectory is for, the must-read entry points (AGENTS, the manual,
the ADR index, storage-v2), and a pointer that design docs/ADRs are
kept-for-history (obsolete ones are not deleted).

## 5. Definition of done

Per AGENTS.md (ruff / mypy / pytest, CHANGELOG, version bump). Plus:
`render_mermaid` unit test on fixture rows; a `--from-tsv` round-trip
test; the generated `schema.md` committed; `docs/README.md` +
`docs/architecture.md` present and linked from the top-level README.

## 6. Non-goals

- No automatic regeneration in CI (manual `scripts/gen-schema` for
  now; a cron/CI hook can come later).
- No ORM / no migration-derived diagram — the **live DB** is the
  source of truth (catches drift between migrations and reality).
- `schema-v2.puml` stays as the hand-drawn *conceptual* sketch with
  its drift note; `schema.md` is the generated *actual*.
